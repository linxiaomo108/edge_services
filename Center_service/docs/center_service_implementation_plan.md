# 中心服务（Center Service）中期优化实施计划

> 文档版本：v1.1（整合 yh.md 优化建议：HLS 时间窗口活跃判断、节点心跳失效、首片超时重启、控制闭环定义）  
> 创建日期：2026-04-03  
> 基于文档：`plan-01.md`（二次评审）、`plan.md`（分析报告）、`yh.md`（优化建议）  
> 核心原则：**解决真实问题，禁止过度优化，单中心单机闭环**

---

## 一、优化目标

构建中心服务，承担以下核心职责：

1. **接收边缘 RTMP 推流** → 通过 ZLMediaKit 转为 HLS
2. **统一播放入口** → 所有客户端通过中心获取 m3u8 播放地址
3. **推流生命周期管理** → 触发 start、超时 stop、失败反控
4. **历史视频集中存储**（可选）→ 接收边缘上传的历史视频

---

## 二、核心约束（红线）

| 编号 | 约束 | 说明 |
|------|------|------|
| R1 | 单中心设计 | 不做分布式，不做集群 |
| R2 | 不引入 MQ | 直接 HTTP 调用边缘接口 |
| R3 | 不做分布式调度 | 单机调度即可 |
| R4 | 不做复杂任务系统 | 推流管理用内存 + 简单数据库 |
| R5 | 不做跨节点状态同步 | 每个边缘节点独立管理 |
| R6 | 中心不感知 RTSP | RTSP 解析由边缘负责，中心只管 stream_id |

---

## 三、架构概览

```text
┌─────────────┐       HTTP        ┌──────────────┐      RTMP       ┌──────────────┐
│   前端/App   │ ───────────────→ │   中心服务    │ ←───────────── │   边缘服务    │
│  (播放 HLS)  │ ← m3u8/ts ───── │ (ZLMediaKit)  │ ──── HTTP ───→ │ (ffmpeg推流)  │
└─────────────┘                   └──────────────┘                 └──────────────┘
                                        │
                                  ┌─────┴─────┐
                                  │  流管理模块  │
                                  │ (生命周期)  │
                                  └───────────┘
```

### 控制面 vs 数据面

| 层面 | 组件 | 职责 |
|------|------|------|
| **控制面（Control Plane）** | 中心服务流管理模块 | start/stop 调度、超时检测、失败反控 |
| **数据面（Data Plane）** | 边缘 ffmpeg → RTMP → ZLMediaKit → HLS | 视频数据传输 |

## 四、技术选型

| 组件 | 选型 | 说明 |
|------|------|------|
| 流媒体服务器 | ZLMediaKit | 开源，支持 RTMP 接收 → HLS 输出，性能优异 |
| 后端框架 | 待定（Java Spring / Python FastAPI / Go） | 需与现有服务端技术栈一致 |
| 数据库 | 复用现有服务端数据库 | 新增推流管理相关表 |
| 部署方式 | 单机部署 ZLMediaKit + 中心服务 | 不做容器化（除非已有基础设施） |

> **注意**：后端框架应与现有服务端（SmartCampus）技术栈保持一致，避免引入额外维护成本。

### 7.3 "请求播放"核心流程

```text
前端请求播放(campusCode, nvrDeviceId, channelNum)
  │
  ├─ 查 stream_registry：该流是否已存在且 state=PUSHING？
  │   ├─ 是 → 直接返回 hls_url（流复用）
  │   └─ 否 → 继续
  │
  ├─ 查 edge_node：找到该校区对应的边缘节点地址
  │
  ├─ 调用边缘 POST /api/stream/open → 获取 streamId
  │
  ├─ 生成 pushUrl = rtmp://{center}/live/{streamId}
  │
  ├─ 调用边缘 POST /api/stream/start(streamId, pushUrl)
  │
  ├─ 等待 ZLMediaKit 确认流注册（轮询或 webhook）
  │
  ├─ 生成 hlsUrl = http://{center}/live/{streamId}/hls.m3u8
  │
  └─ 写入 stream_registry，返回 hlsUrl
```

### 7.4 stream_registry 数据模型

| 字段 | 类型 | 说明 |
|------|------|------|
| id | bigint | 主键 |
| stream_id | varchar(128) | 流唯一标识（边缘生成） |
| edge_node_id | varchar(64) | 边缘节点标识 |
| campus_code | varchar(64) | 校区编码 |
| nvr_device_id | int | NVR 设备 ID |
| channel_num | int | 通道号 |
| state | varchar(16) | INIT / PUSHING / STOPPED / FAILED |
| push_url | varchar(256) | RTMP 推流地址 |
| hls_url | varchar(256) | HLS 播放地址 |
| created_at | datetime | 创建时间 |
| updated_at | datetime | 最后更新时间 |
| last_active_at | datetime | 最后活跃时间（有人播放） |
| viewer_timeout_sec | int | 无人观看超时秒数，默认 60 |

## 九、控制闭环定义（系统稳定性的基础）

### 9.0 正常闭环

```text
[用户播放]
   ↓
服务端 → 中心 play-request
   ↓
(无缓存) → open → edge
   ↓
中心 start
   ↓
edge ffmpeg 推流 → RTMP → ZLMediaKit
   ↓
ZLM 生成 HLS
   ↓
用户播放 m3u8
   ↓
(无访问 60s+，时间窗口判断)
   ↓
中心 stop
   ↓
edge 停止推流
```

### 9.0.1 异常闭环

**情况1：推流失败（ffmpeg 启动失败或中途断流）**

```text
ZLM 无流 / 断流
→ 中心检测（轮询或 webhook）
→ start(force=True)
→ 仍失败 → stop + 标记 FAILED
```

**情况2：edge 假死（推流进程无响应）**

```text
中心检测断流
→ stop
→ start(force=True)
→ 仍失败 → 标记 FAILED
```

**情况3：无人观看**

```text
60s+ 无 m3u8/ts 请求 且 无前端心跳
→ 中心 stop
→ edge 停止推流
→ 释放资源
```

**情况4：HLS 首片超时（摄像头 GOP 大 / 网络抖动）**

```text
start 后 15 秒无 HLS 首片
→ stop
→ start(force=True)
→ 仍失败 → 标记 FAILED
```

> **关键**：以上四条闭环必须全部实现，否则系统会进入"卡死 / 资源泄漏 / 假活"状态。

| 4.1 | 实现超时回收定时任务 | 定时扫描 `stream_registry`：`last_active_at` 超过 60 秒 → 调用边缘 stop | ⬜ 未开始 | | | |
| 4.2 | ZLMediaKit 流断开检测 | 监听 ZLMediaKit webhook 或轮询流列表，检测流是否断开 | ⬜ 未开始 | | | |
| 4.3 | 失败反控：流断开 → 调用边缘 stop | ZLMediaKit 报告流断开 → 中心调用边缘 `/api/stream/stop` → 更新状态 | ⬜ 未开始 | | | |
| 4.4 | HLS 首片超时自动重启机制 | start 后启动计时器（默认 15 秒），如果超时后 ZLMediaKit 仍未收到 HLS 首片 → 调用边缘 stop → 调用 start(force=True) 重试 → 仍失败则标记 FAILED。超时时间可通过配置 `HLS_FIRST_CHUNK_TIMEOUT_SEC` 调整（摄像头 GOP 大时可设为 20~30 秒） | ⬜ 未开始 | | | |
| 4.5 | 失败反控：推流失败 → 可选 force 重启 | 中心检测到流异常 → 调用 `start(force=True)` 尝试恢复 → 仍失败则 stop | ⬜ 未开始 | | | |
| 4.6 | 资源泄漏兜底检查 | 定期对比 ZLMediaKit 活跃流 与 stream_registry 记录，清理孤儿流 | ⬜ 未开始 | | | |
| 4.7 | 验证：无人观看超时自动停止 | 播放 → 关闭页面 → 60 秒后流自动停止 | ⬜ 未开始 | | | |
| 4.8 | 验证：推流失败自动恢复 | 模拟边缘网络断开 → 中心检测 → force 重启 → 恢复推流 | ⬜ 未开始 | | | |

### 9.3 超时回收流程

```text
定时任务（每 10 秒执行一次）
  │
  ├─ 遍历 stream_registry 中 state=PUSHING 的记录
  │
  ├─ 检查 last_active_at：
  │   ├─ now - last_active_at > viewer_timeout_sec（默认 60s）
  │   │   └─ 调用边缘 POST /api/stream/stop(streamId)
  │   │       └─ 更新 state = STOPPED
  │   └─ 否 → 跳过
  │
  └─ 结束
```

### 9.4 失败反控流程

```text
ZLMediaKit 报告流断开（webhook 或轮询）
  │
  ├─ 查 stream_registry：该 stream_id 是否 state=PUSHING？
  │   ├─ 否 → 忽略（可能是正常 stop）
  │   └─ 是 → 异常断流
  │
  ├─ 第一次异常：调用边缘 start(force=True) 尝试恢复
  │   ├─ 成功 → 等待 HLS 重新生成
  │   └─ 失败 → 继续
  │
  ├─ 第二次异常：调用边缘 stop → state = FAILED
  │
  └─ 记录失败日志，通知管理员（可选）
```

## 十一、不合理之处与解决方案

### 11.1 ⚠️ ZLMediaKit 的 HLS 延迟问题

**问题**：HLS 天然有分片延迟（通常 6~10 秒），对于"实时"监控场景可能不满足需求。

**解决方案**：
- HLS 分片时长配置为 2~3 秒，列表保留 3 个分片，理论延迟 6~9 秒
- 如果延迟不可接受，后续可考虑 ZLMediaKit 的 WebSocket-FLV 或 HTTP-FLV 低延迟方案
- **当前阶段先做 HLS，验证链路通畅后再优化延迟**

### 11.2 ⚠️ 中心单点故障

**问题**：单中心设计，如果中心宕机则所有实时流不可用。

**解决方案**：
- **当前阶段不解决**，这是设计约束（R1：单中心设计）
- 保留边缘 play 接口作为兜底，中心不可用时可临时直连边缘
- 后续根据实际需求决定是否做高可用

### 11.3 ⚠️ ZLMediaKit 与中心服务的部署耦合

**问题**：ZLMediaKit 和中心服务是否部署在同一台机器？

**解决方案**：
- **推荐同机部署**：减少网络跳转，简化运维
- 中心服务通过 `127.0.0.1:1935` 访问 ZLMediaKit
- 对外只暴露中心服务的 HTTP 端口和 ZLMediaKit 的 HLS 端口

### 11.4 ⚠️ 多校区场景下的推流地址路由

**问题**：如果有多个校区（多个边缘节点），中心需要知道每个边缘节点的网络可达地址。如果边缘 IP 变了、VPN 断开、节点离线，中心还在用旧地址调用，会出现"幽灵节点"问题。

**解决方案**：
- 复用现有 `registerInfo` 机制，边缘注册时上报自己的可达地址
- 中心维护 `edge_node` → `address` 映射
- 调用边缘接口时使用注册的地址
- **必须补充心跳失效机制**：

```text
边缘每 30 秒发送心跳 → 中心/服务端记录 last_seen_at
中心定时检查：now - last_seen_at > 90秒 → 标记节点不可用
调用边缘前检查节点状态 → 不可用则跳过并返回错误
节点恢复心跳 → 自动标记可用
```

> **关键**：没有心跳失效机制，系统会出现"幽灵节点"——地址失效但中心不知道，这是最难排查的问题之一。

### 11.5 ⚠️ 中心如何判断"有人在看"

**问题**：中心需要知道 HLS 是否还有人在播放，才能决定是否超时 stop。方案中提到"60 秒无人播放则 stop"，但如何判断"无人播放"？

> **关键认识**：HLS 是 HTTP 短连接分片请求，用户不保持长连接。因此**瞬时连接数 ≠ 用户数**，不能用 ZLMediaKit 当前连接数来判断是否有人在看，否则会出现"用户在看 → 此刻无请求 → 被误判为无人观看 → 中心误 stop → 播放中断"的问题。

**解决方案：时间窗口活跃判断（三信号组合）**

| 信号 | 作用 | 实现方式 |
|------|------|----------|
| **ZLM 请求时间窗口**（主判断） | 判断最近 N 秒内是否有 m3u8 / ts 请求 | ZLMediaKit hook 或日志记录最后请求时间，检查 `now - last_request_at < 30s` |
| **前端心跳**（兜底） | 弥补 ZLM 统计盲区 | 前端每 30 秒发心跳，中心更新 `last_active_at` |
| **stop 延迟**（防误杀） | 避免瞬时无请求导致误停 | 超时阈值设为 60 秒以上，确保至少跨越 2 个 HLS 分片周期 |

**判断逻辑**：

```text
满足以下任一条件 → 视为"有人在看"：
  1. ZLM 最近 30 秒内有 m3u8/ts 请求
  2. 前端最近 60 秒内有心跳

两个条件均不满足，且持续超过 60 秒 → 触发 stop
```

> **注意**：不再使用 ZLMediaKit 瞬时连接数作为判断依据，改为基于请求时间窗口的活跃度判断。

### 11.6 ⚠️ 同型号 NVR 的快速探索策略无法由中心统一下发

**问题**：当前边缘侧的通道快速探索规则主要写在客户端代码内，设备级缓存也只存在本地库。即使某个型号在一批客户端上已经积累出稳定的高概率探索顺序，中心服务也无法把这套“型号级探索策略”统一下发给所有客户端，导致规则优化依赖客户端发版传播。

**目标边界**：
- 中心共享的是**型号级探索策略模板**，不是某台具体设备的成功映射结果
- 中心不下发“通道 1 固定等于 33”这类绝对结论，只下发优先级规则、扫描范围、启用开关等策略信息
- 边缘运行时仍需结合 `startChan`、`startDChan`、`ipChanNum` 和本地设备级缓存做最终决策

**解决方案（纳入中期优化，不立即实施）**：
- 中心新增“型号级探索策略”配置能力，支持按 canonical model 管理候选优先级规则
- 中心新增策略版本号与更新时间管理，客户端只在版本变化时拉取完整策略，避免高频全量同步
- 中心提供策略启用、停用、回滚能力，避免错误规则一次性影响所有客户端
- 中心与边缘的职责边界明确为：**中心负责策略发布与版本控制，边缘负责策略落地与运行时组合决策**

| 序号 | 补充任务 | 具体内容 | 完成状态 | 完成时间 | 自测结果 | 自测时间 |
|------|----------|----------|----------|----------|----------|----------|
| fix-6.1 | 设计型号级探索策略数据模型 | 新增策略表，字段建议包含 `model`, `strategy_json`, `version`, `enabled`, `updated_at` | ⬜ 未开始 | | | |
| fix-6.2 | 新增策略查询接口 | 提供边缘拉取型号级探索策略的接口，支持按版本增量判断 | ⬜ 未开始 | | | |

## 十三、依赖与风险

| 风险项 | 影响 | 缓解措施 |
|--------|------|----------|
| ZLMediaKit 编译/部署困难 | 阶段一延期 | 提前准备编译环境，或使用预编译版本 |
| 边缘到中心网络不通 | 推流失败 | 确保 RTMP 端口可达，VPN 路由正确 |
| HLS 延迟过高 | 用户体验差 | 调小分片时长，后续可切 FLV 低延迟 |
| ZLMediaKit 不稳定 | 流中断 | 失败反控 + force 重启机制 |
| 多校区地址路由 | 中心找不到边缘 | 复用 registerInfo 注册机制 |
