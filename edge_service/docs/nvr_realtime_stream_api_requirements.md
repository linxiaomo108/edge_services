# 实时视频流接口需求分析文档

## 1. 需求背景与目标

### 1.1 背景

当前部署形态中：

- **服务端部署在公网**
- **NVR 位于各校区内网**
- **公网服务端无法直接访问 NVR**
- **边缘服务部署在校区内网，可与 NVR 直连**
- **边缘服务所在机器后续会做公网 IP/端口映射**
- **服务端通过映射后的公网地址访问边缘服务接口**
- **当前开发阶段也可能处于纯内网环境，此时服务端可直接通过内网地址访问边缘服务接口**

因此，需要在现有边缘服务基础上新增一个**独立的实时视频流接口能力**，由边缘服务承担：

- 校区侧接入点
- NVR 通道解析与管理
- 实时视频流中转
- 基础校区隔离校验

### 1.2 目标

本需求的目标是：

- **新增独立接口**
  - 由边缘服务对外提供“获取指定 NVR 指定通道实时视频流”的能力

- **支持校区隔离**
  - 请求必须带 `campusCode`
  - 边缘服务需校验该值是否与本校区配置一致

- **流程尽量简单**
  - 服务端尽量少调用接口
  - 不引入复杂多级鉴权、多轮握手、多次跳转
  - 同时兼容**开发阶段内网访问**与**部署阶段公网映射访问**

- **支持后续维护与扩展**
  - 本地维护 NVR 与通道信息
  - 支持默认规则匹配通道
  - 支持候选通道优先匹配
  - 支持后续逐步扩展到更多设备厂商或更多流格式

---

## 2. 需求范围

### 2.1 本次纳入范围

- 独立实时视频流接口设计
- 接口增加 `campusCode` 参数
- 校区一致性校验
- NVR 与通道信息的本地维护
- 内网直连与公网映射地址配置设计
- 实时流返回方式设计
- 开发跟踪文档结构设计

### 2.2 本次不强制纳入范围

为保持流程简单，本阶段**不建议强制加入**：

- 多重访问令牌校验
- 多次接口握手
- 复杂权限模型
- 高并发限流体系
- 多租户/多角色授权体系
- WebRTC 等高复杂实时协议

---

## 3. 设计原则

### 3.1 简单易用

服务端为了播放一个视频流，应尽量只做**一次主要请求**，避免：

- 先取 token
- 再申请 session
- 再轮询状态
- 再获取播放地址

这种流程虽然更完整，但当前会明显提高接入复杂度与推广成本。

在访问入口上，也应尽量保持统一：

- **开发阶段**优先支持内网地址直连
- **部署阶段**支持公网映射地址访问
- 无论内网还是公网，都尽量复用同一套接口协议与返回结构

### 3.2 保留基础安全，不做过度设计

本阶段建议只保留必要安全控制：

- `campusCode` 校验
- 可选的基础来源校验或简单共享密钥
- 请求参数完整性校验

不建议引入过重方案。

### 3.3 独立实现，不影响现有功能

新增接口必须：

- 独立于现有任务接口
- 不影响现有下载、转码、语音、监控逻辑
- 尽量复用已有底层能力，但不耦合现有任务流

### 3.4 可持续扩展

接口设计要允许未来平滑扩展：

- 多 NVR
- 多通道
- 多厂商
- 多输出协议
- 本地缓存优化

---

## 4. 核心功能与流程描述

## 4.1 核心功能概述

边缘服务新增一套**实时流中转能力**，服务端提交指定通道参数后，边缘服务完成：

- 校区校验
- 通道解析
- 流地址构建
- 拉流/中转
- 向服务端返回可播放或可读取的实时视频流

---

## 4.2 推荐的服务端调用方式

### 推荐方案：单主接口获取播放地址

为了简化服务端接入，推荐优先采用：

- **服务端直接请求一个实时流接口**
- 边缘服务内部完成所有校验、解析、通道匹配与流地址准备
- 成功后返回 `publicPlayUrl` 与 `lanPlayUrl`
- 调用方根据自身部署环境选择可访问地址，边缘服务不再基于来源 IP 自动切换

### 不推荐的复杂方式

当前阶段不建议强制采用以下复杂流程：

- 创建 session
- 查询 session
- 轮询启动状态
- 再调用播放地址
- 再做 token 二次交换

除非后续确实遇到并发、复用或性能问题，再升级为多接口方案。

---

## 4.3 推荐接口模式

### 模式 B：单接口返回播放 URL

#### 接口职责

- 接收参数
- 校验 `campusCode`
- 创建或复用中转会话
- 返回 `publicPlayUrl` 与 `lanPlayUrl`

#### 优点

- 服务端只多一步“使用返回 URL 播放”
- 更适合后续做会话复用与资源管理
- 更容易同时兼容**内网直连**与**公网映射**两种访问环境
- 更方便排查地址拼接、映射配置和代理状态问题

#### 建议

如果后续考虑会话复用、空闲回收、诊断追踪，同时又希望保持调用方式简单，**模式 B 更利于扩展且更适合当前阶段统一落地**。

模式 A 与“直接输出流”是对应关系，模式 B 与“返回播放地址”是对应关系。  
**本需求当前统一推荐采用模式 B：单接口返回 `publicPlayUrl` 与 `lanPlayUrl`。**

统一推荐模式 B 的原因：

- **调用仍然简单**
  - 服务端只需先请求一次接口，再直接使用返回地址播放或拉流
- **同时兼容内网与公网**
  - `publicPlayUrl` 可用于公网映射或默认回退地址
  - `lanPlayUrl` 可用于同校区或已确认可直连边缘服务内网的场景
- **更适合后续扩展**
  - 更容易接入会话复用、诊断、日志和过期控制
- **更利于联调与排障**
  - 服务端拿到明确地址后，更方便核对访问路径和环境配置

---

## 4.4 推荐流程

### 流程一：服务端请求实时视频流

- **[1]** 服务端调用边缘服务实时流接口
- **[2]** 传入以下参数：
  - `campusCode`
  - `nvrDeviceId`
  - `nvrChannelNum`
  - `nvrChannelId`
  - `ipAddress`
  - `port`
  - `account`
  - `password`
  - 可选候选通道参数
- **[3]** 边缘服务校验 `campusCode`
- **[4]** 校验通过后，查询本地 NVR/通道缓存
- **[5]** 若命中缓存，则直接使用缓存解析结果
- **[6]** 若未命中，则按默认规则或候选通道进行通道匹配
- **[7]** 构造 RTSP 源地址
- **[8]** 启动实时流代理/中转，并生成 `publicPlayUrl` 与 `lanPlayUrl`
- **[9]** 返回地址规则：
  - `publicPlayUrl` 优先返回公网映射地址，若未配置映射则回退为边缘服务本机地址
  - `lanPlayUrl` 返回边缘服务本机局域网地址
  - 不再基于请求来源 IP 自动切换内外网地址
- **[10]** 调用方根据自身部署环境选择使用 `publicPlayUrl` 或 `lanPlayUrl`

---

### 流程二：`campusCode` 校验流程

- **[1]** 从请求参数读取 `campusCode`
- **[2]** 获取边缘服务本地配置中的校区编码
- **[3]** 比较两者是否一致
- **[4]** 若不一致：
  - 直接返回错误
  - 不继续执行后续 NVR 访问、通道匹配或拉流逻辑
- **[5]** 若一致：
  - 进入后续处理流程

---

### 流程三：通道解析流程

- **[1]** 优先使用请求中显式传入的候选通道参数
- **[2]** 若候选参数未命中，再使用本地默认规则匹配
- **[3]** 成功后写入本地 SQLite 映射缓存
- **[4]** 下次优先复用缓存结果，减少探测耗时

---

## 5. 接口设计要点

## 5.1 请求参数设计

### 必填参数

- `campusCode`
  - 当前新增
  - 用于校区隔离校验

- `nvrDeviceId`
  - NVR 设备 ID

- `nvrChannelNum`
  - NVR 设备通道号

- `nvrChannelId`
  - NVR 设备通道 ID

- `ipAddress`
  - NVR 内网 IP 地址

- `port`
  - NVR 端口
  - 默认 `554`

- `account`
  - NVR 登录账号

- `password`
  - NVR 登录密码

### 可选参数

- `candidateChannels`
  - 候选通道列表
  - 用于减少通道探测耗时

- `streamProfile`
  - 码流类型
  - 例如 `main` / `sub`

- `outputProtocol`
  - 输出协议
  - 例如 `flv` / `mpegts`

- `reuseIfExists`
  - 是否复用已有流会话

---

## 5.2 `campusCode` 校验设计

### 目标

实现最基础、最直接的校区隔离控制，避免：

- 请求误打到别的校区边缘服务
- 服务端误连错误校区
- 外部错误调用进入处理流程

### 设计要求

- 请求参数中必须带 `campusCode`
- 边缘服务启动时读取本地校区编码配置
- 每次请求先做一致性比对
- 校验失败立即终止

### 错误返回建议

可返回以下语义之一：

- `非法校区访问`
- `campusCode 校验失败`
- `请求校区与边缘服务所属校区不一致`

### 设计建议

该校验应放在整个流程最前面，优先级高于：

- 参数深入解析
- NVR 登录
- 通道探测
- ffmpeg 拉流

---

## 5.3 简单播放流程设计

### 目标

让服务端尽量做到：

- 传一次参数
- 直接拿到可播放地址

### 推荐实现方向

- **优先方案**
  - 一个主接口完成校验、通道解析、播放地址返回

- **次优方案**
  - 一个接口直接返回流内容

### 统一建议

为消除模式与返回方式上的歧义，本需求统一建议如下：

- **推荐接口模式**
  - 模式 B：单接口返回播放 URL

- **推荐返回方式**
  - 返回 `publicPlayUrl` 与 `lanPlayUrl`

- **两者对应关系**
  - 模式 A 对应“直接输出流”
  - 模式 B 对应“返回播放地址”

- **本阶段最终推荐**
  - 模式 B + 返回 `publicPlayUrl` 与 `lanPlayUrl`

### 不建议

当前阶段不建议将一个播放动作拆成以下复杂链路：

- 预检查接口
- 校区校验接口
- 通道解析接口
- 创建 session 接口
- 获取播放地址接口
- 心跳续租接口

这些设计在当前阶段会增加复杂度，偏离“快速落地”目标。

---

## 5.4 NVR 通道管理设计

### 目标

边缘服务需自动维护当前校区内所有 NVR 及其通道基础信息。

### 实现思路

建议本地 SQLite 维护以下几类数据：

- NVR 基础信息
- 业务通道信息
- 通道解析结果
- 流代理会话信息

### 管理原则

- 优先读取历史成功映射
- 未命中时再探测
- 探测成功后立即缓存
- 支持“默认规则 + 候选通道参数”的双轨机制

---

## 5.5 公网映射配置设计

### 目标

边缘服务需要根据所处环境，返回**当前服务端真实可访问的地址**。该地址不应只支持公网，也应支持开发阶段的内网直连访问。

### 配置入口路径

建议按以下方式统一配置：

- **客户端启动配置**
  - 路径：`config.json`
  - 用途：作为客户端正式配置入口，适合打包后和日常部署使用

- **开发环境配置页/配置文件**
  - 路径：`E:\Project\SmartCampus-school\edge_services\output\monitor_config.json`
  - 用途：开发环境下可作为监控页配置落地点，逻辑与 `config.json` 保持一致

- **优先级说明**
  - 当前系统中，`config.json` 的对应字段会覆盖 `monitor_config.json` 的同名字段
  - 因此如果两个文件同时配置，最终以 `config.json` 中的值为准

### 建议配置项

建议增加独立配置段，逻辑上包含：

- `enableStreamProxy`
  - **用途**：控制是否启用实时流中转能力
  - **是否必需**：建议保留，但通常只需要改这一个开关
  - **说明**：这是功能总开关。建议默认关闭；启用实时流代理时将其改为 `true`

- `publicBaseUrl`
  - **用途**：直接指定边缘服务对外暴露的完整基础地址，例如 `https://x.x.x.x:19080`
  - **是否必需**：公网部署场景下强烈建议优先配置；纯内网开发场景可不配置
  - **说明**：若配置了该项，建议优先使用它拼接返回地址，因为最直接、最不容易出错；默认留空

- `publicHost`
  - **用途**：指定映射后的公网主机名或 IP
  - **是否必需**：不是绝对必需
  - **说明**：当未配置 `publicBaseUrl` 时，可与 `publicScheme`、`publicPort` 组合生成公网地址；默认留空

- `publicPort`
  - **用途**：指定映射后的公网端口
  - **是否必需**：不是绝对必需
  - **说明**：仅在需要拼接公网地址且端口非默认值时非常有用；默认留空

- `publicScheme`
  - **用途**：指定公网访问协议，例如 `http` 或 `https`
  - **是否必需**：不是绝对必需
  - **说明**：当通过分项拼接公网地址时需要；若已配置 `publicBaseUrl`，则可弱化该项的重要性；默认留空

### 默认值与生效规则

- **默认规则**
  - `publicHost / publicPort / publicScheme / publicBaseUrl` 默认都为空字符串
  - 空值表示当前未显式配置公网映射地址

- **空值行为**
  - 当公网映射参数为空时，系统按**内网环境**处理，返回边缘服务在局域网内可访问的地址

- **已配置行为**
  - 当配置了 `publicBaseUrl`，或配置了 `publicHost` 且可以拼出完整地址时，系统自动按**公网映射环境**处理，优先返回公网地址

### 注释写法说明

- **JSON 限制**
  - `config.json` 与 `monitor_config.json` 都是 JSON 文件，不能使用原生注释语法

- **建议做法**
  - 使用 `_comment`、`_comment_xxx` 这类说明字段保存配置注释
  - 这样既不破坏 JSON 结构，也便于开发阶段直接查看说明

- **落地要求**
  - `config.json` 中建议放入 `streamProxy` 段及 `_comment` 说明字段
  - `output/monitor_config.json` 中也可保持相同结构，逻辑一致

### 最少需要改哪几个参数

- **纯内网开发环境**
  - 只需要把 `enableStreamProxy` 改成 `true`

- **公网映射环境（推荐方式）**
  - 把 `enableStreamProxy` 改成 `true`
  - 配置 `publicBaseUrl`
  - 其余 `publicHost / publicPort / publicScheme` 可以继续留空

- **公网映射环境（备用方式）**
  - 把 `enableStreamProxy` 改成 `true`
  - 至少配置 `publicHost`
  - 再按实际情况补充 `publicPort` 与 `publicScheme`

### 设计说明

边缘服务内部监听地址与最终返回地址可能不同：

- **内部地址**
  - 边缘服务实际监听的本地地址
  - 开发阶段通常可直接作为返回地址使用

- **外部地址**
  - 公网 IP/映射端口
  - 部署阶段通常优先作为返回地址使用

接口响应中若需要返回播放地址，建议按以下优先级生成：

- **优先级 1**
  - 若配置了 `publicBaseUrl`，优先返回基于 `publicBaseUrl` 拼接的地址

- **优先级 2**
  - 若未配置 `publicBaseUrl`，但配置了 `publicHost`，则使用 `publicScheme + publicHost + publicPort` 拼接公网地址

- **优先级 3**
  - 若当前为纯内网开发环境且未配置公网映射参数，则返回边缘服务内网可访问地址

## 5.6 返回格式设计

### 推荐返回方式一：返回播放 URL

适合服务端自己再发起播放请求。

返回内容建议包括：

- `success`
- `message`
- `campusCode`
- `nvrDeviceId`
- `nvrChannelNum`
- `nvrChannelId`
- `publicPlayUrl`
- `lanPlayUrl`
- `outputProtocol`
- `resolvedChannel`
- `sourceType`

### 地址返回说明

地址字段建议按以下规则返回：

- **`publicPlayUrl`**
  - 若配置了 `publicBaseUrl` / `publicHost` / `publicPort`，优先返回公网映射地址
  - 若未配置映射地址，则回退为边缘服务本机地址
- **`lanPlayUrl`**
  - 返回边缘服务本机局域网地址
- **统一原则**
  - 边缘服务不再猜测调用方所处网络，不再基于来源 IP 自动切换地址
  - 由调用方根据自身网络环境选择真正可访问的地址

## 6. 关键实现方式说明

## 6.1 模块拆分建议

建议新增独立模块，职责如下：

- **接口层**
  - 接收请求、参数校验、返回响应

- **校区校验层**
  - 专门处理 `campusCode` 比对

- **NVR 通道管理层**
  - 管理设备、通道、解析缓存

- **流代理层**
  - 负责 RTSP 拉流、中转输出、会话管理

- **配置层**
  - 管理公网映射地址、开关、超时配置

---

## 6.2 可复用的现有基础能力

当前系统中已经具备一定复用基础：

- Web 接口入口
- SQLite 封装
- NVR 通道映射持久化能力
- 海康 RTSP 路径构造能力
- ffmpeg 调用能力

因此实现上应采用：

- **复用底层能力**
- **新增独立业务模块**
- **不改现有核心任务流**

---

## 6.3 数据持久化建议

### 建议维护的数据对象

- **NVR 设备信息**
- **通道信息**
- **通道解析结果**
- **流会话信息**

### 持久化目的

- 降低重复探测成本
- 提高流请求响应速度
- 记录上次成功通道
- 支持后续排障和跟踪

## 7. 合理假设说明

## 10. 建议的最终方案结论

### 结论概述

本需求最适合采用以下方案：

- 新增一个**独立的实时视频流接口**
- 请求中增加必填 `campusCode`
- 接口收到请求后先做**校区一致性校验**
- 校验通过后，再进行：
  - NVR 参数处理
  - 通道匹配
  - 拉流中转
  - 生成并返回 `publicPlayUrl` 与 `lanPlayUrl`
- 全流程坚持“**简单易用、快速落地**”原则
- 不引入复杂多次鉴权或多接口串联流程
- 使用本地 SQLite 维护 NVR、通道和映射缓存
- 同时支持**开发阶段内网地址返回**与**部署阶段公网映射地址返回**
- 优先采用**模式 B：单接口返回 `publicPlayUrl` 与 `lanPlayUrl`**

### 当前最推荐的落地方向

- **接口层面**
  - 单主接口优先
  - 统一采用模式 B：单接口返回 `publicPlayUrl` 与 `lanPlayUrl`

- **安全层面**
  - 先做 `campusCode` 基础校验
  - 不做过度复杂安全链路

- **能力层面**
  - 边缘服务作为实时流代理与校区侧接入点
  - 返回地址需兼容内网与公网两类环境

- **文档层面**
  - 用本文档作为后续开发与测试的统一跟踪清单

总结：**在新增 `campusCode` 校验后，整套方案应围绕“校区隔离 + 简单调用 + 独立接口 + 本地通道管理 + 双环境地址返回”这五个核心目标展开，统一推荐采用“模式 B：单接口返回 `publicPlayUrl` 与 `lanPlayUrl`”的方式，优先保障快速接入与稳定落地。**
